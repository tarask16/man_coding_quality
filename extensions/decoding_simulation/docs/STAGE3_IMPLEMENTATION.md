# Реализация этапа 3

## 1. Вход и выход

Входом является материальное сообщение

\[
C=(c_1,c_2,\ldots,c_n),
\]

сформированное на этапе 2. Выходом служит нормативный план

\[
D_h(C)=(d_1,d_2,\ldots,d_n),
\]

где каждому фактически присутствующему элементу `c_i` соответствует один
шаг обратной процедуры `d_i`.

## 2. Синтаксис нормативного токена

Поддерживается абстрактный формат:

```text
TOK_<encoding_operation>_<source_position>_<encoded_value>
```

Разбор выполняется регулярным выражением с фиксированной шириной позиции.
Операция может содержать символы подчеркивания. Значение извлекается только
из наблюдаемого токена.

## 3. Правила обратных операций

| Прямая операция | Обратная операция |
|---|---|
| `abstract_substitution` | `abstract_inverse_substitution` |
| `abstract_numeric_mapping` | `abstract_inverse_numeric_mapping` |
| `abstract_service_marking` | `abstract_inverse_service_interpretation` |
| `abstract_copying` | `abstract_inverse_copying` |

Для неизвестного токена применяется диагностическое правило
`abstract_unresolved_inverse_operation`. Оно не создает кандидатное значение.

## 4. Защита от скрытой подстановки

`EncodedElement` содержит трассировочное поле `source_value`, однако
`DecodingProcedureModel` его не читает. Это предотвращает искусственное
восстановление исходного сообщения при поврежденном токене. В метаданных
плана фиксируется:

```text
uses_source_value_for_decoding = false
```

## 5. Порядок элементов

Шаги формируются строго в порядке `encoded_position`. Извлеченное из токена
`declared_source_position` сохраняется как диагностический атрибут и не
используется для автоматической перестановки. Поэтому позиционная ошибка
этапа кодирования остается наблюдаемой в дальнейшем сквозном моделировании.

## 6. Граница этапа

На этапе 3 не создается `DecodedMessage` и не рассчитывается расстояние между
`M` и `M'`. Эти функции появятся после подключения модели декодирующего
оператора, ошибок и контроля декодирования.
