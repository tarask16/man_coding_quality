# Этап 2. Материальное представление кодированного сообщения C

## Назначение

Этап вводит явные типы `EncodedElement`, `EncodedMessage`, `EncodingTraceStep`,
`EncodingProtocol` и `MaterialEncodingResult`. Нормативные токены
`ProcedureStep.abstract_token` преобразуются в упорядоченную материальную
последовательность `C`.

## Источники данных

- `ProcedurePlan` задает нормативную последовательность;
- `ControlProtocol` определяет, какие ошибки остались после контроля;
- изменяется только новая рабочая копия C;
- объекты базового пакета не модифицируются.

## Семантика остаточных ошибок

| Тип базовой ошибки | Изменение C |
|---|---|
| `abstract_substitution_error` | детерминированная замена токена |
| `abstract_omission_error` | отсутствие элемента в C с сохранением трассы |
| `abstract_position_error` | смещение элемента в последовательности |
| `abstract_reference_error` | ошибочный абстрактный токен обращения |
| `abstract_service_marker_error` | ошибочный служебный токен |
| неизвестный тип | маркированная абстрактная замена |

Исправленная контролем ошибка (`residual_error=False`) не изменяет C.

## Инварианты

1. Без остаточных ошибок `C` совпадает с нормативной последовательностью.
2. Каждый нормативный шаг представлен в `EncodingProtocol`, включая пропуски.
3. Идентификаторы `message_id`, `procedure_id`, `step_id` проверяются.
4. Базовые `ProcedurePlan` и `ControlProtocol` остаются неизменными.
5. Выходные файлы создаются только внутри `extensions/decoding_simulation/`.

## Ограничение этапа

Этап не выполняет обратное декодирование и не формирует `M_prime`. Формальная
обратная процедура реализуется только на этапе 3.
