# Реализация этапа 4

## 1. Архитектурная роль

Этап 4 добавляет детерминированный слой между формальным планом
`D_h(C)` и будущей вероятностной моделью ошибок декодирования.

```text
EncodedMessage C
    -> DecodingPlan D_h(C)
    -> DecodingOperatorPlanEstimate O_d
    -> DecodingConditionPlanEstimate U_d
```

Новый слой находится только в пакете `manual_coding_sim_decoding`.

## 2. Отдельный декодирующий оператор

`DecodingOperatorProfile` не наследует и не изменяет базовый
`OperatorProfile`. Это позволяет моделировать случаи, когда кодирование и
декодирование выполняют разные лица.

Профиль включает:

- подготовленность;
- опыт;
- базовое и минимальное внимание;
- скорость утомления;
- навык работы со справочными материалами;
- навык интерпретации кодовых элементов;
- относительную скорость работы.

`DecodingOperatorModel` рассчитывает только ожидаемые величины. Наличие
низкого внимания не означает, что ошибка уже возникла.

## 3. Отдельные условия декодирования

`DecodingConditionProfile` задает условия `U_d`, независимые от условий
кодирования:

- лимит времени;
- шум;
- доступность инструкции;
- рабочую нагрузку;
- частоту прерываний;
- качество освещения.

Модель корректирует время и внимание и рассчитывает индекс устойчивости.

## 4. Неразрешенные токены

Неразрешенный входной токен увеличивает `ambiguity_pressure`, но не
исправляется и не получает кандидатное значение. Это сохраняет результат
этапа 3 и не создает скрытого доступа к исходному сообщению.

## 5. Контрольные признаки

В выходном JSON явно фиксируются:

```text
uses_only_decoding_plan = true
uses_source_value_for_decoding = false
generates_decoding_errors = false
creates_decoded_message = false
```

## 6. Следующий этап

Этап 5 должен ввести вероятностную модель ошибок декодирования на основе
полученных оценок внимания, утомления, сложности, неоднозначности и условий.
