# Этап 3. Формальная обратная процедура декодирования

## Назначение

Этап вводит нормативный план `D_h(C)` поверх материального кодированного
сообщения `EncodedMessage`. Модель описывает последовательность обратных
операций, но пока не моделирует действия оператора, ошибки декодирования,
контроль декодирования и восстановленное сообщение `M'`.

## Архитектурные ограничения

- базовый пакет `src/manual_coding_sim` не изменяется;
- новые и измененные файлы находятся только в
  `extensions/decoding_simulation`;
- фактический порядок элементов `C` сохраняется;
- встроенная в нормативный токен позиция используется только для диагностики;
- поле `EncodedElement.source_value` не используется для получения
  кандидатного результата;
- поврежденный или неизвестный токен фиксируется как неразрешенный шаг.

## Основные классы

- `TokenParseResult`;
- `DecodingStep`;
- `DecodingPlan`;
- `DecodingProcedureModel`;
- `DecodingOperationRule`;
- `FormalDecodingConfig`.

## Результат CLI

```text
extensions/decoding_simulation/reports/stage3/decoding_plan_demo.json
```

## Критерий завершения

1. Контракт базового пакета совместим.
2. Все нормативные токены стандартного прогона распознаются.
3. Неизвестные токены не восстанавливаются по `source_value`.
4. Тест этапа 3, регрессия расширения и полная регрессия проекта проходят
   локально.
